# OpenClaw Debugging Log - April 9, 2026

**Session Focus:** Fixing Telegram Group Unresponsiveness and AgentMail Plugin Integration.

## 1. Telegram Group Unresponsiveness

### Symptoms
- Bots added to the "Team DF" group chat ignored all messages and @-mentions.
- Bots were confirmed Admins with privacy mode disabled.
- Logs showed no incoming message events being processed by the agents.

### Diagnostic Steps
- Created a standalone Python script (`telegram_poll.py`) to bypass OpenClaw and verify Telegram's API delivery. Confirmed that the API *was* delivering group messages to the bot token.
- Modified `openclaw.json` to set `groupPolicy: "open"`.
- Tested both `requireMention: true` and `false`.

### Root Cause & Discoveries
- **The Wildcard Bug**: Found that OpenClaw's internal router sometimes fails to match the `"*"` wildcard in `channels.telegram.groups` against specific group chat IDs. If the group ID isn't explicitly listed, the event is dropped silently before reaching the agent.
- **The Connection Wake**: Telegram sometimes requires an explicit command like `/start@BotUsername` to be sent in the group to "handshake" the presence of the bot.

### Resolution
- Explicitly added the group ID `-5182129954` to the `groups` object in `openclaw.json`.
- Sent `/start@JooJJBot` and `/start@DF_Sysop_Bot` in the group chat.
- **Result:** Both bots now respond to @-mentions instantly in the group.

## 2. Startup Delays & The Bonjour Watchdog

### Symptoms
- The container reported "Up" but the Telegram provider would not start for 35–45 seconds.
- Test messages sent immediately after `docker restart` were lost.

### Discovery
- **The Boot Delay Loop**: The `openclaw-agentmail-listener` plugin had a malformed configuration (missing `apiKey`).
- **Mechanism**: Every 500ms, the plugin attempted to connect and failed. This repeated failure caused the Bonjour service watchdog to hold the "Ready" status until a timeout was reached.

### Resolution
- Temporarily disabled the plugin to restore 2-second boot speeds, then fixed the configuration schema (see below).

## 3. AgentMail Plugin Configuration

### Symptoms
- Plugin reported `invalid config: apiKey: must have required property 'apiKey', inboxId: must have required property 'inboxId'`.

### Discovery
- **Schema Errors**: The configuration was attempted at the top level and inside an `extensions` key. 
- **Correct Path**: In this OpenClaw version, plugin config must be located at `plugins.entries.<plugin-id>.config`.
- **Key Name Mismatch**: The schema required `inboxId` (string), but instructions had used `inboxes` (array).

### Resolution
- Moved config to the correct block:
  ```json
  "openclaw-agentmail-listener": {
    "enabled": true,
    "config": {
      "apiKey": "am_us_...",
      "inboxId": "excitedmagazine226@agentmail.to",
      "sessionKey": "agent:main:telegram:direct:8682401511"
    }
  }
  ```

## 4. AgentMail Delivery & Automation

### Symptoms
- Notifications arrived in the group chat instead of the private DM, or were duplicated.
- The `PATCH` call to mark emails as read was failing with 404/400 errors.

### Diagnosis
- **Session Routing**: The plugin defaulted to the heartbeat session (`agent:main:main`), which routed output to the *last active* chat (the group).
- **ID Formatting**: The `message_id` from AgentMail includes angle brackets (`<...>`), which were breaking the URL path in the `curl` PATCH call.

### Resolution
- **Delivery Fix**: Updated `sessionKey` in `openclaw.json` to point to the user’s direct Telegram ID (`8682401511`).
- **Script Fix**: Created `check-agentmail.sh` with Python URL-encoding (`urllib.parse.quote`) to handle the brackets in `message_id`.
- **Duplication Fix**:
  1. Modified the script to send directly to Telegram via the Bot API.
  2. Removed the `delivery` block from the OpenClaw native cron job to prevent the LLM from redundantly forwarding the same output.
- **Result:** Now receives a single, clean notification in the private DM every hour (or instantly via WebSocket).
